# FileWeft 0.0.2（开发候选说明）

当前仓库版本：`0.0.2-SNAPSHOT`

> 本文件描述当前开发线准备进入 `0.0.2` 的变更与升级边界，不代表 `0.0.2` 已正式发布，也不证明任何制品已经上传到 CNB 或其他 Maven 仓库。

从本开发线开始，Maven group 统一为 `ai.icen`，JVM 包名统一为 `ai.icen.fw`。正式版本发布前仍须完成仓库定义的全部 `releaseCheck` 门禁，并由项目所有者确认许可证、安全披露入口、制品签名和目标仓库策略。

## 数据库迁移命名空间

- FileWeft 迁移脚本只位于 `classpath:ai/icen/fw/db/migration`，迁移历史只写入 `fileweft_schema_history`，不再占用宿主通常使用的 `classpath:db/migration` 和 `flyway_schema_history`。
- Starter 的迁移模式为 `DISABLED`、`VALIDATE` 或 `MIGRATE`，默认 `DISABLED`。生产环境应由受控 Migration Job 执行 `MIGRATE`，API/Worker 使用 `VALIDATE`；Starter 不负责让宿主进程在迁移后自动退出。
- `fileweft.persistence.schema` 必须与 DataSource 的 `SELECT current_schema()` 一致。目标 schema 只有在 `MIGRATE + create-schema=true` 时允许创建；生产环境推荐预建 schema 并保持 `create-schema=false`。
- 新库首次初始化会写入受严格校验的版本 `0` 命名空间标记，然后仍从 `V001` 开始执行全部业务迁移。缺失、重复或被篡改的标记、未来版本、旧 FileWeft 共享 history 记录，以及没有专属 history 的既有 FileWeft 表都会使启动失败。
- 迁移资源在发布门禁中按名称、数量和字节内容核对，并拒绝 JAR 中重新出现旧的 `db/migration` 路径。

## Spring Boot 启动安全

- Boot 2 与 Boot 3 Starter 使用一致的显式迁移属性，并在启用全局 lazy initialization 时仍会主动校验配置和执行所选迁移模式。
- FileWeft 初始化器声明数据库初始化依赖，确保宿主数据库初始化先完成；宿主 Flyway 与 FileWeft Flyway 保持独立 history 和 location。
- 只有一个 DataSource 时 Starter 才会自动创建 `FlywayMigrationRunner`。存在多个 DataSource 时，即使其中一个标记为 `@Primary`，也必须由宿主显式提供绑定正确 DataSource 的 runner；当模式为 `VALIDATE` 或 `MIGRATE` 时，零个或多个 runner 都会失败关闭。
- `VALIDATE` 不创建或修改数据库对象；`create-schema=true` 只允许与 `MIGRATE` 组合。自定义 runner 或 initializer 仍优先于默认 Bean。

## 开发拓扑

仓库内 `fw-dev` Compose 将 API 设为 `MIGRATE`，Worker 设为 `VALIDATE` 并等待 API 健康；两者都通过 JDBC `currentSchema=fileweft_dev` 使用同一 FileWeft schema。模拟下游平台以 `platform` profile 启动，通过独立 JDBC `currentSchema=fileweft_dev_platform` 和自身迁移器维护下游数据，不加入 FileWeft 迁移命名空间。

## 与 0.0.1 的关系

历史 `com.fileweft:*:0.0.1` 已在项目中标记为撤回，不得用于新接入。撤回状态、17 个历史坐标和旧库处理边界见 [`0.0.1` 历史记录](0.0.1.md)。该状态不表示 CNB、镜像或本地缓存中的副本已经物理删除；当前代码也不会自动收养 `0.0.1` 可能写入的共享 Flyway history。

当前接入方式、权限要求和旧库人工迁移流程分别以仓库根 `README.md` 与 `docs/production-operations.md` 为准。
