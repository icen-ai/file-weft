# FileWeft 0.0.2

> **历史发布说明：** 本页原样记录 FileWeft `0.0.2` 的发布事实；这些事实不因
> 产品更名而改变。当前开发合同与产品决策已由 [FlowWeft 根开发合同](../../AGENTS.md)、
> [1.0 产品范围 ADR](../decisions/0001-flowweft-1.0-product-scope.md)、
> [更名 ADR](../decisions/0002-flowweft-product-rename.md) 和
> [通用工作流 ADR](../decisions/0003-generic-workflow-platform.md) 取代。

状态：`v0.0.2` 稳定发布说明。

本文件记录 `v0.0.2` 的精确发布合同。`ai.icen:*:0.0.2` 只有在受保护标签流水线全部成功，且远端公开仓库能够被匿名冷缓存独立消费者重新解析后才可作为稳定制品消费；本说明不以文字提前声称该远端证据已经成功，源码、标签或本地制品也不能替代它。

## 已确定的产品范围

### Agent 默认关闭

`0.0.2` 不提供 Agent 产品能力。Agent 将重新设计，但无限期延后，最早只能在
`1.0.0` 正式发布后重新评估；这不构成对 `1.x`、下一版本或任何版本的交付承诺。
为避免破坏已经发布的调用方和数据库，`fileweft-agent`、Agent SPI/公共 ABI 以及
V012/V026 中相关表和列继续保留，但仅用于兼容。默认运行时、Starter、Doctor/
插件清单、正式 HTTP 和 `fileweft-dev` 均不注册、不宣传、不展示也不验收 Agent。
只有维护既有 0.0.1 接入时才可显式启用
`fileweft.compatibility.legacy-agent-autoconfiguration-enabled=true`；该开关不是新功能。

### 正式目录树 HTTP 移出本版本

独立的正式 catalog/目录树 HTTP 资源已明确移出 `0.0.2`，没有承诺目标版本，也
不是本次发布的隐藏阻断项。既有 `DocumentCatalogProvider` SPI、目录绑定以及
catalog-aware 读取、下载和生命周期授权 guard 继续是受支持的宿主集成边界；
“没有正式目录树 HTTP”不等于取消这些安全能力。

## 0.0.2 发布内容

### 正式 HTTP 与运行时

- 新增正式五操作断点续传资源：创建会话、读取权威检查点、流式 PUT 分片、同步完成与放弃，统一位于 `/fileweft/v1/uploads`。完成返回稳定 `uploadId/fileObjectId/fileAssetId` 回执，不隐式创建文档。
- 续传会话按可信租户和 owner 隔离；创建 key 只保存租户域摘要。staging、`QUARANTINED`、完成围栏、提交结果未知对账和确定拒绝后的原子 reset 防止跨用户接管、误删已完成对象或把含糊结果当作失败。
- Boot 2 与 Boot 3 提供镜像的正式 MVC、错误外层、缓存安全头与负面协议行为；formal-v1 浏览器验收覆盖审批后 REQUIRED/OPTIONAL 多目标交付、断点恢复、授权与双租户隔离。
- 新增受权工作流决策证据读取；新审批/驳回不可变保存操作者 ID、可选安全显示名与时间，遗留任务保持 `UNKNOWN`，普通审批历史继续身份脱敏。

### 数据库与迁移

- PostgreSQL、MySQL、KingbaseES 当前均具有完整的 28 个 V001–V028 迁移资源。`verifyFileWeftMigrationResources` 校验发布 JAR 中方言资源的完整性、唯一性和源文件字节身份。
- MySQL 支持边界仅为原生 MySQL 8.x 的 8.0.17+，本轮实库迁移/JDBC repository 证据版本为 8.0.46；不包含 MariaDB 或 MySQL 9，也不把该证据扩大到所有小版本、排序规则或拓扑。KingbaseES 实证版本为 V008R006C009B0014。
- V017 在数据库层保证每个租户/文档至多一个 `PENDING` 工作流：PostgreSQL/KingbaseES 使用部分唯一索引，MySQL 使用 nullable stored generated columns 加唯一索引。
- V026 扩宽宿主身份列并增加工作流决策证据。它不猜测遗留操作者，对 Agent 表的同等更新仅用于数据库兼容。
- V027 为 Outbox 与 Task 增加 `(created_time, id)` Worker claim-order 索引；MySQL repository 使用对应索引提示，避免 `SKIP LOCKED` 前锁住过多候选行。
- V028 将全部 18 张 FileWeft 自有 MySQL 业务表转换为 NO PAD 的 `utf8mb4_0900_bin`，让 tenant、用户、业务、任务、存储、幂等及其他 opaque ID 按 Unicode 标量/文本值精确比较，大小写、重音和尾空格均不折叠。显示文本也采用同一排序规则，这是防止未来 key 列继承会折叠标识的默认值所作的安全优先取舍；PostgreSQL/KingbaseES V028 仅保持版本对齐。
- `FlywayMigrationRunner` 已验证 Spring Boot 2 管理的 Flyway 8.5.13、FileWeft 自身的 9.22.3 与 Spring Boot 3 管理的 11.7.2。Boot 3 的 `flyway-core`、`flyway-mysql`、`flyway-database-postgresql` 必须同为 11.7.2。
- Kingbase Starter 默认只包装 Spring Boot 已选给 Flyway 的 DataSource，应用主 DataSource 仍是真实 Kingbase。只有宿主具备等价实库集成时，才可关闭 `fileweft.persistence.kingbase-flyway-compatibility-enabled`。

### 消费者、制品与发布验证

- 独立 release-smoke 消费者从待发布 Maven 仓库解析全部 17 个公共模块；Boot 2 与 Boot 3 消费者均由宿主显式提供 `spring-boot-starter-jdbc`、连接池与 H2，启动真实 Web 上下文并请求健康端点，纯库消费者验证 SPI/兼容制品可解析编译。
- Boot 2 消费者显式对齐 FileWeft 的 Kotlin 2.1.21 运行时契约，并验证 Boot 2 管理的 SLF4J 1.7.36；文档同步给出 Gradle、Maven 与 dependencyInsight 检查方式。
- CycloneDX SBOM 以 17 个发布模块各自的生产 `runtimeClasspath` 可达闭包生成，排除 Dev、Gradle build environment 和测试/编译孤儿。
- 发布仓库与 ZIP 门禁核对精确模块库存、POM/module metadata、SNAPSHOT/stable 身份、checksum、许可证/NOTICE、危险或重复 JAR entry、SBOM 及独立消费者；错误、混合构建或路径穿越夹具必须失败。

### 按需本地验证与 CNB

- 本地开发先运行受影响测试类和模块任务，一批连贯改动只补一次 `fastCheck`；普通开发不默认展开跨 JDK、真实数据库、RustFS、浏览器或发布门禁。
- PR 与 `main` 按变更路径并行选择 fast、文档、Java 兼容、PostgreSQL、MySQL、KingbaseES、RustFS、Dev 验收和 release artifact lane；`.dockerignore`、Gradle 锁、release-smoke、LICENSE/NOTICE 等关键路径有不得选择零 lane 的契约测试。
- 夜间与稳定标签运行完整矩阵。标签发布聚合精确 40 位提交 SHA、事件与分支证据，发布失败也销毁 CNB token；发布后使用隔离且清空的 Gradle User Home 从公开 CNB Maven 仓库做匿名冷缓存回读。

## 升级与回滚边界

- 作为历史边界，`v0.0.1` 只发布了 PostgreSQL V001–V025，没有发布 MySQL、KingbaseES 或 V026–V028。受保护 `v0.0.2` 标签成功发布后，三种方言的 V001–V028 全部成为不可改写的稳定迁移资源；后续版本仍须只前进并保持 checksum 可审计，禁止用无条件 `flyway repair` 掩盖差异。
- V026 升级先运行 `docs/sql/postgresql-v026-workflow-decision-evidence-preflight.sql`，停止旧审批入口并等待在途事务；应用回滚保留身份列、约束和证据。
- V027 使用普通建索引；V028 在 MySQL 上可能重建全部表和文本索引。两者都要求停写维护窗口、可恢复备份、足够的原表/重建/索引/临时/redo/binlog/复制磁盘预算，以及锁与复制监控。
- V027 失败后检查并清理无效或定义不符的同名索引再前进重试，不伪造 history。应用回滚保留 V027 索引和 V028 NO PAD `utf8mb4_0900_bin`；绝不能回到 `utf8mb4_bin`、`*_ci` 或其他 PAD SPACE/折叠比较，否则可能折叠只在大小写、重音或尾空格上不同的 tenant/opaque ID，造成越权命中、错误唯一冲突或幂等身份变化。
- `0.0.1` 的 Maven 坐标、源码标签和发布 ZIP 保持不可变。

## 发布身份与可消费条件

本文件对应 `v0.0.2`，但不能单凭这份说明判定远端制品已经可用。只有精确提交取得全部应触发 CNB lane 的成功证据、受保护标签发布成功，并从远端公开仓库完成匿名冷缓存消费者验收后，`ai.icen:*:0.0.2` 才具备稳定可消费状态；任一步失败都必须按未完成发布处理。
