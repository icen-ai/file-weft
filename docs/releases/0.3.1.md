# FileWeft 0.3.1

状态：`v0.3.1` 候选发布合同。

本文件记录 `0.3.1` 的目标产品与兼容边界，不提前证明远端已经发布。只有受发布门禁约束的 `v0.3.1` 标签匹配受保护发布分支（`main` 或维护分支 `ver/0.3`）HEAD、精确提交取得全部应触发 CNB lane 的成功证据，并从公开仓库以全新隔离缓存匿名回读全部 19 个坐标后，`ai.icen:*:0.3.1` 才可作为稳定制品消费；源码、本文、本地构建或部分绿灯均不能替代这些证据。

## 版本定位

`0.3.1` 是 `0.0.3` 产品线的第一个补丁版本。版本号沿用三段 `X.Y.Z` 语义：`0.0.3` 实质是 `0.3.0`，自本版本起产品线记为 `0.3.x` 并从维护分支 `ver/0.3` 发布，不再要求补丁版本合并或回退 `main`。

本版本只交付真实政企内网集成（天津水务 `waterx-transpond`）实证暴露的可观测性与契约缺陷修复，不改变模块库存、数据库迁移序列或公共 ABI：

- 正式发布模块保持 19 个不变；
- 三种数据库方言的迁移保持 V001–V029 不变，不追加、不改写任何既有迁移或 checksum；
- 所有既有公共构造器、方法与 SPI 签名保持二进制兼容，新增能力一律以新增构造器、重载或可空字段表达。

## 投递链路可观测性（P0 修复）

- `OutboxWorker` 接入 `FileWeftLogger` SPI：Worker 启动输出一行就绪日志（workerId、轮询间隔、批次大小、最大尝试次数）；每个非空批次输出 claimed/succeeded/retried/failed/lost 摘要，含失败批次升级为告警；事件处理失败以 error 级输出并携带原始异常与租户/Trace 上下文。空轮询不产生日志，未注入 logger 时行为与 `0.0.3` 完全一致。
- Outbox `last_error` 持久化内容从"仅异常类名"扩展为"异常类名 + 消息"，消息中的换行压平为单行，总长度仍截断至既有 1024 上限。
- `ResilientFileConnector` 不再丢弃底层异常：调用与健康检查失败以 error 级记录原始异常；熔断器打开、半开探测失败与恢复、线程池/队列饱和拒绝分别输出 warn/info/debug 日志并携带 connectorId。返回给调用方并持久化的诊断文案只附带异常类名，不拼接下游敏感消息全文。
- API 进程新增 Outbox 积压提醒：当本进程未启用 Worker 且持久化 Outbox 存在待处理事件时，周期性输出 WARN 提醒部署 Worker；提醒按积压计数变化节流，积压排空后重新武装，不依赖任何指标后端。

## 契约修复（真实集成反馈）

- `submit` 成功回执现在携带 `taskId`：单审路由返回唯一待办 task，多人会签路由返回第一个待办 task。持久化幂等重放只保存文档与工作流标识，重放回执的 `taskId` 为 `null`；客户端在任何重试路径都应容忍该字段为 `null` 并回退到待办查询。
- 同一文档的重复 `submit`（即使换用新 `Idempotency-Key`）不再返回冲突：提交对单文档天然幂等（数据库层唯一活跃工作流索引保证），现在返回指向现有 `PENDING` 工作流的同一回执，且不重复写入提交审计。损坏状态（非 `PENDING`、跨文档、跨租户）与路由解析竞态仍按原语义冲突；publish、approve、reject、withdraw 等其它入口的冲突语义完全不变。
- `sync-status` 的交付目标新增可空 `lastErrorCategory` 字段：失败或重试中的目标附带固定枚举类别（`TARGET_NOT_FOUND` / `DOCUMENT_UNAVAILABLE` / `FENCE_INVALID` / `CONNECTOR_FAILURE` / `STORAGE_FAILURE` / `PERSISTENCE_FAILURE` / `UNKNOWN`），成功或无诊断时为 `null`。持久化的原始错误文本可能含主机策略、存储或数据库细节，永不进入 HTTP 契约；该字段是纯增量可空字段，向后兼容。

## 发布机制

- 发布身份门禁从"发布提交必须等于远端 `main` HEAD"扩展为"必须等于受保护发布分支 HEAD：`main` 或该标签所属 `X.Y` 产品线的维护分支 `ver/X.Y`"。两条来源至少匹配一条，均不匹配（含分支不存在）失败关闭。维护分支只追加、不回溯，发布前必须在 CNB 为实际发布来源分支启用分支保护。

## 兼容与升级说明

- 从 `0.0.3` 升级到 `0.3.1` 不需要任何迁移、配置或代码改动。
- 依赖"重复 submit 返回 409"的客户端需要按新语义调整：重复提交返回 `200` 与现有工作流回执。依赖首次 `submit` 回执 `taskId` 为 `null` 的客户端应按文档改为接受字符串或 `null`。
- Boot 2 与 Boot 3 Starter 的 bean 名称、装配条件与默认 Agent 不暴露边界保持不变。

## 可消费条件

`0.3.1` 的代码完成、文档完成和本地测试都不等于远端发布完成。只有受发布门禁约束的 `v0.3.1` 标签匹配受保护发布分支 HEAD、对精确提交完成全部十一条验证 lane、受信发布步骤成功、发布凭据已销毁，并使用隔离且清空的 Gradle User Home 从公开 CNB Maven 仓库匿名解析全部 19 个坐标及 Boot 2、Boot 3、纯 SPI 消费者后，才能宣布 `ai.icen:*:0.3.1` 稳定可消费。
