# FileWeft 0.3.2

状态：`v0.3.2` 候选发布合同。

本文件记录 `0.3.2` 的目标产品与兼容边界，不提前证明远端已经发布。`0.3.2` 的可消费条件与 `0.3.1` 一致：受发布门禁约束的标签匹配受保护发布分支（`main` 或维护分支 `ver/0.3`）HEAD、精确提交完成全部必需 lane、受信发布步骤成功、发布凭据已销毁，并从公开仓库以隔离且清空的缓存匿名回读全部 19 个坐标及消费者。源码、本文、本地构建或部分绿灯均不能替代这些证据。

## 版本定位

`0.3.2` 是 `0.3.x` 产品线的第二个补丁版本，从维护分支 `ver/0.3` 发布。本版本交付设计评审（`docs/design-review-0.0.3.md`）与真实集成反馈确认的可诊断性、契约一致性与 SPI 卫生修复，不改变正式发布模块库存（19 个）。

## 幂等重放回执等价（V030）

- `fw_idempotency_record` 新增可空 `result_subresource_id varchar(256)` 列（三种数据库方言各一份 V030 迁移；PostgreSQL/Kingbase 以 `NOT VALID` + `VALIDATE CONSTRAINT` 两阶段重建结果约束，MySQL 只加列）。V030 为纯追加可空列，可与 `0.3.1` 旧节点滚动混跑，不需要停写窗口。
- `IdempotencyResult` 增加尾部可选 `subresourceId` 槽（`@JvmOverloads`，既有构造器重载不变）。submit 落库首个待办 task 标识后，同一 `Idempotency-Key` 的持久化重放返回与首次完全一致的回执（含 `taskId`）。
- `0.3.1` 及更早落库的历史记录没有该列，其重放回执的 `taskId` 仍为 `null`；客户端在任何重试路径都应容忍 `taskId` 为 `null` 并回退到待办查询。approve/reject/withdraw 的幂等结果语义不变。

## 可诊断性与弹性

- 插件与内置 DoctorChecker 统一按单 checker 5 秒预算执行（可覆盖），超时与执行器饱和映射为固定安全 ERROR 结果，不泄漏线程细节；permission checker 保持直通。
- S3 存储客户端增加 `apiCallTimeout`（默认 2 分钟）与 `apiCallAttemptTimeout`（默认 30 秒）上限及显式标准重试策略；dev 宿主可通过 `fileweft.storage.api-call-timeout-millis` / `api-call-attempt-timeout-millis` 覆盖。
- metadata schema 的 `format` 正则按字符串缓存编译（有界 512，溢出降级不缓存），128 字段写入从最多 256 次编译降为每 format 一次；非法 format 的异常类型、消息与抛出层不变。
- metadata 配置异常全部保留原始 cause；`DocumentMetadataService` 可注入 `FileWeftLogger` 对配置类失败记 error（不注入则静默，行为与 `0.3.1` 一致）。`JdbcApplicationTransaction` 的 cleanup 日志从 JUL 迁到 `FileWeftLogger` SPI。

## Boot 3 bean 名恢复（0.0.3 回归修复）

- Boot 3 Starter 的 94 个 bean 恢复与 Boot 2 一致的 `fileWeft*` 主名；`0.0.3` 的短名（`transaction`、`documents`、`workflows` 等）保留为**别名**，已适配短名的宿主不受影响，宿主自定义同名 bean 不再触发 override 冲突。短名别名已标记废弃，将在后续大版本移除。

## SPI 卫生（含两项有意的兼容决策）

- `DocumentRepository.findForMutation` / `findByDocumentNumber` 的默认实现已删除并移入新加法子接口 `DocumentMutationRepository`。依赖默认 `findByDocumentNumber`（返回 `null`，**静默关闭文档号唯一性约束**）或默认 `findForMutation`（运行时抛 `UnsupportedOperationException`）的自定义实现不再可用：文档写路径静态要求 `DocumentMutationRepository`；Starter 在能力缺失时启动期明确失败，catalog 加法能力按既有语义不装配。自定义持久化实现方需重新编译并实现新接口；默认 `JdbcDocumentRepository` 已实现。
- 新增业务异常 `WorkflowTaskDeniedException`（403）与 `WorkflowTaskMissingException`（404），框架抛出点已切换；`WorkflowTaskAssignmentDeniedException`、`WorkflowTaskNotFoundException` 标记 `@Deprecated` 并改为继承新基类。**行为变化（有意修复）**：框架抛出的工作流任务异常不再是 `SecurityException`/`NoSuchElementException`，宿主按 JDK 类型捕获的代码需要迁移到新基类；HTTP 契约不变（仍 403/404 固定文案）。
- `@Deprecated` 标记（将在后续大版本移除，本版本不删）：`MetadataValue`、`core/result` 的 `Result`/`ErrorDetail`/`ErrorCode`/`FileWeftException`（未被各层采用的保留模型）、两套 `FileWeftRuntimeConfiguration`（僵尸 facade）、web 双 starter 中无路由映射的 `create`/`addVersion` 方法（注明非 HTTP 端点）。
- `DoctorCheckerContractTest` 合约补强：不抛异常、结果确定性、evidence 敏感词元、输出有界等 7 项不变量，既有实现方继承基类即获得。

## 发布机制与工程卫生

- 修复 Flyway 宿主兼容测试任务在开启 configuration cache 时导致构建失败的问题（`notCompatibleWithConfigurationCache`，与发布身份任务同款处理）。
- 新增根 `.gitattributes`：迁移 SQL 按字节冻结（`*.sql -text`），杜绝行尾"修复"改写已发布迁移 checksum。
- `.ci/README.md` 新增契约变更检查单：改动正式 v1 DTO/回执/错误码/路由形状必须同 PR 同步 `fileweft-dev/web/e2e` 精确断言，`devAcceptanceCheck` 为必需证据。

## 兼容与升级说明

- 数据库：升级前按方言执行 V030 迁移（`migration-mode=MIGRATE`）；纯追加可空列，可与 `0.3.1` 节点滚动混跑。回滚保留该列，不清空已知值。
- 从 `0.3.1` 升级无需配置改动；自定义持久化实现与按 JDK 类型捕获工作流异常的宿主需按上文"SPI 卫生"调整并重新编译。
- Boot 2 / Boot 3 Starter 的装配条件、默认 Agent 不暴露边界保持不变。
