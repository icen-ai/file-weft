# FileWeft 0.0.3

状态：`v0.0.3` 候选发布合同。

本文件记录 `0.0.3` 的目标产品与兼容边界，不提前证明远端已经发布。只有受发布门禁约束的 `v0.0.3` 标签匹配受保护远端 `main` HEAD、精确提交取得全部应触发 CNB lane 的成功证据，并从公开仓库以全新隔离缓存匿名回读全部 19 个坐标后，`ai.icen:*:0.0.3` 才可作为稳定制品消费；源码、本文、本地构建或部分绿灯均不能替代这些证据。

## 发布定位

`0.0.3` 是 metadata 扩展与生命周期加固版本，在保持公共兼容的前提下交付：

- 通用、版本化的 metadata schema 契约与运行时；
- 文档创建/新增版本的 schema 校验、规范化及安全 HTTP 投影；
- Metadata Doctor；
- 待审工作流撤回；
- 断点续传与 Spring Runtime 配置的内部职责拆分。

本版本不增加 Agent 产品能力、厂商适配器、业务专用模型或架构重写。`0.0.2` 与 `0.0.3` 均不提供 Agent；既有 `fileweft-agent`、Agent SPI/公共 ABI 和 V012/V026 中相关数据库形状仅为源码、二进制与数据库兼容保留，默认运行时、Starter、Doctor/插件清单、正式 HTTP 与 `fileweft-dev` 不注册、宣传或验收该能力。Agent 最早只能在 `1.0.0` 正式发布后重新评估，且没有承诺版本。

## Metadata schema

### 模块与契约

正式发布模块由 17 个增加到 19 个，新增：

- `fileweft-metadata-api`：纯 JDK 8、Java 友好的不可变契约，不依赖 Spring、JDBC、HTTP、ORM 或厂商 SDK；
- `fileweft-metadata-runtime`：纯 JVM 的 schema 注册、解析、校验、规范化与处理实现。

现有资产持久化的 `Map<String, String>` 兼容形状保持不变，metadata 层是加法扩展，不把业务专用对象写入 Core、Domain 或 SPI。schema 由稳定 `id`、`version` 和字段规则构成；字段类型固定为 `STRING`、`NUMBER`、`BOOLEAN`、`DATE`、`ENUM` 与 `STRING_LIST`。运行时支持必填、类型、枚举、长度和格式校验，并对结果进行规范化。

`MetadataSchemaRegistry` 同时支持当前版本与精确历史版本解析。默认 Starter 把普通 `MetadataSchema` Bean 视为当前版本，把 `HistoricalMetadataSchema` Bean 视为保留的精确历史版本；自定义 `MetadataSchemaResolver` 存在时默认注册表整体退让。租户只从可信 `TenantProvider` 上下文取得，请求不能提交或覆盖；处理流程固定为 authorize → schema resolve → exact-version recheck → validate → normalize → persist，Controller 不直接持久化原始 metadata。

### Application、HTTP 与 Doctor

- `GET /fileweft/v1/metadata/schemas/{schemaId}` 返回当前租户可解析 schema 的 `id`、`version` 和字段安全规则，不暴露租户、实现类、注册来源或内部配置。
- Schema 与 HTTP metadata 输入统一最多 128 个字段；`format` 使用 RE2/J 线性时间正则方言并在注册时拒绝不支持的回溯构造，避免由长输入触发正则拒绝服务。
- `POST /fileweft/v1/documents` 与 `POST /fileweft/v1/documents/{documentId}/versions` 可选携带恰好一个 `metadataSchemaId` 及重复的 `metadata` multipart part；每个值使用 `field=value`。输入在授权后、文件写入和持久化前完成 schema 校验与规范化，并在文档资产保存 `metadata.schema-id` / `metadata.schema-version` 标记；新增版本完整替换文档级 schema metadata，但保留目录等框架键。
- schema 业务 metadata 不复制到厂商对象 user-metadata header；FileWeft 数据库中的 `FileAsset.metadata` 是权威副本，避免对象存储配额或不可读回行为造成“校验成功但不可诊断”。
- 不携带 metadata 时保持既有写入行为；一旦携带 metadata 而 schema/处理能力未安全装配，则失败关闭，不能退化为未校验 Map。
- `MetadataDoctorChecker` 检查 schema 缺失、字段无效及历史 metadata 与精确 schema 版本不兼容，只返回固定、安全诊断，不回显内部异常或原始敏感值。

## 审批撤回

新增 `LifecycleCommand.WITHDRAW_REVIEW` 与正式路由：

```text
POST /fileweft/v1/workflows/{workflowId}/withdraw
```

合法转换为文档 `PENDING_REVIEW -> DRAFT`、工作流 `PENDING -> WITHDRAWN`。只有可信提交者或通过 `document:review:withdraw` 授权的操作者可以执行；已完成工作流、跨租户、隐藏目录文档和未授权调用均失败关闭。目录模式仍在事务外取得短期 ACL 决策，并在最终写入前重新验证文档可见性。

撤回必须携带持久化 `Idempotency-Key`。每次重放重新认证和授权；最终事务按 idempotency → document → asset（目录模式）→ workflow 的既有锁序，原子保存文档/工作流状态、审计和稳定回执。撤回与最后一次 approve/reject 只能有一个串行结果，不能用旧聚合快照覆盖对方。

V029 为 `fw_workflow_instance` 新增可空 `submitted_by varchar(256)`。新提交保存可信当前用户；历史工作流没有不可歧义的提交者证据，迁移故意保留 `NULL`，不得从 task assignee、审计、当前用户目录或幂等记录猜测回填。历史空值只能由策略授权的操作者撤回。PostgreSQL 在数据库层验证宿主用户 ID 安全约束；MySQL 与 KingbaseES 延续既有兼容策略，在可信应用边界验证。

## 内部可维护性

公共 `ResumableUploadService` 构造器与方法继续作为兼容 facade，现有上传、所有者隔离、分片确认、完成围栏、对账、放弃和清理语义不变。实现内部将职责拆给 `ResumableUploadStarter`、`ResumableUploadPartHandler`、`ResumableUploadCompletionHandler`、`ResumableUploadAbortHandler`、`ResumableUploadReconciler` 与 `ResumableUploadCleanupService`，由 facade 组装这些协作者并共享受限的安全上下文；这些协作者不是新的公共 API。

Boot 2 与 Boot 3 的大体量 Runtime wiring 按 `FileWeftDocumentConfiguration`、`FileWeftUploadConfiguration`、`FileWeftWorkflowConfiguration`、`FileWeftDoctorConfiguration` 与 `FileWeftDeliveryConfiguration` 分担，保留原条件装配、客户 Bean 覆盖优先级和默认 Agent 不暴露边界。这是内部组织调整，不授权宿主依赖内部配置类作为新的稳定 ABI。

## 发布库存与迁移历史

- 当前正式 Maven/SBOM/远端消费者库存为 19 个模块；新增的两个 metadata 模块必须出现在发布仓库、POM/module metadata、SBOM 以及匿名冷缓存回读中。
- 当前 PostgreSQL、MySQL 与 KingbaseES 各有 29 个迁移 V001–V029；发布 JAR 必须只在 `ai/icen/fw/db/migration` 包含完整、唯一且与源 SQL 字节一致的三套资源。
- 历史事实保持不变：`v0.0.1` 只发布 PostgreSQL V001–V025；`v0.0.2` 对应三种方言的 V001–V028。`0.0.3` 只追加 V029，不改写任何既有迁移或 checksum。

V029 不能与旧审批写节点滚动混跑。升级时先关闭 submit/approve/reject/withdraw，停止旧 API 节点并等待在途审批事务完成，再执行 V029、核验专属 `fileweft_schema_history` 和列/约束，最后启动 `0.0.3` 节点。紧急应用回滚保留 V029 列与已写入的提交者证据，不清空已知值、不删除列、不猜测历史提交者；旧二进制运行期间保持审批写入口关闭，直至恢复支持 `WITHDRAWN` 的节点并完成对账。

## 按需验证与 CNB 完成条件

日常开发按变更范围取最窄证据并做并集：先运行受影响测试类，再运行受影响模块的 `test`，一批连贯代码或构建逻辑改动后只运行一次 `fastCheck`。普通开发不默认运行无模块限定的 `test`/`check`、`compatibilityCheck`、全部外部系统或发布门禁，也不使用 `clean`、`--rerun-tasks`、`--no-build-cache`、`--no-daemon` 破坏增量状态。同一 checkout 同时只运行一个 Gradle wrapper；独立 CNB runner 与真正隔离的 worktree 可以并行。

外部证据按边界追加且失败关闭：

- V029/JDBC/persistence 需要 `postgresIntegrationCheck`；MySQL 与 Kingbase 方言分别还需要 `mysqlIntegrationCheck`、`kingbaseIntegrationCheck`，不能用 PostgreSQL 代替；
- metadata 与撤回的公共 API/Runtime/Starter 需要对应聚焦测试、模块测试、`fastCheck` 和受影响 Java 8/17 lane；若 Dev API/UI/Compose 行为被纳入，还需要 `devAcceptanceCheck`；
- 19 模块库存、POM/metadata、SBOM 与发布 bundle 需要 `releaseArtifactCheck`；稳定标签再运行完整 release 验证和远端匿名消费者回读。

CNB 证据必须匹配交付提交的完整 40 位 SHA，以及正确的 `event`、`sourceRef`、`targetRef` 和 SN；完成要求总体 `status=success`、`pipelineFailCount=0`，并且对应路径规则应触发的全部业务 pipeline 均存在且成功。旧 SHA、运行中、取消、部分绿灯或不相关事件均不是完成证据。诊断失败时先读取首个失败业务 stage，再决定是否改代码；修复后的新提交必须用新 SHA/SN 重新闭环。只读 CLI 流程见 [`.ci/README.md`](../../.ci/README.md)。

## 可消费条件

`0.0.3` 的代码完成、文档完成和本地测试都不等于远端发布完成。只有受发布门禁约束的标签匹配受保护远端 `main` HEAD、对精确提交完成全部必需 lane、受信发布步骤成功、发布凭据已销毁，并使用隔离且清空的 Gradle User Home 从公开 CNB Maven 仓库匿名解析全部 19 个坐标及 Boot 2、Boot 3、纯 SPI 消费者后，才能宣布 `ai.icen:*:0.0.3` 稳定可消费。
