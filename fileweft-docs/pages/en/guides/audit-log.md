Regulated teams need more than a document store—they need a trail. FileWeft records document-level events such as lifecycle transitions, workflow decisions, delivery attempts, and sync results. This page shows how to query that trail without violating tenant isolation or leaking identity into metrics.

## 1. What is logged

The audit log captures events generated by FileWeft itself, not by host business code. Typical entries include:

| Event category | Examples |
| --- | --- |
| Lifecycle | Draft created, submitted for review, published, offlined, archived, restored. |
| Workflow decisions | Task approved, task rejected. |
| Delivery | Connector sync succeeded, sync failed, removal retry requested. |
| Tasks | OCR started, OCR succeeded, OCR permanently failed. |
| System | Doctor check run, cleanup completed. |

The log is append-only from FileWeft's perspective. Hosts should not write directly into FileWeft audit tables; instead, emit domain events to your own audit store if you need richer business context.

## 2. Query document logs

The stable endpoint returns a chronological list for one document:

```bash
curl http://localhost:8080/fileweft/v1/documents/DOC-001/logs
```

The response follows the standard FileWeft v1 envelope and contains a chronological list of events for the document. Use the trace ID to correlate with host gateway logs.

> [!NOTE]
> The exact field names and ordering are subject to the 0.0.1 response contract. See the [HTTP API reference](../reference/http-api.md) for the current shape.

## 3. Read workflow decisions

For documents that went through review, the dedicated workflow-decisions endpoint shows the outcome of each approval task:

```bash
curl http://localhost:8080/fileweft/v1/documents/DOC-001/workflow-decisions
```

This view requires both `document:read` and `document:audit` permissions. In 0.0.1 it returns task outcomes and decision timestamps. Detailed operator identity snapshots are planned for a future release and are not promised as stable behavior.

## 4. Tenant and privacy boundaries

Audit queries are always scoped to the current tenant resolved by the `TenantProvider`. Cross-tenant access is rejected at the SPI layer, not by a request parameter.

FileWeft metrics intentionally exclude high-cardinality or sensitive labels:

- No `tenantId` label on counters or gauges.
- No document ID label.
- No user ID label.

If you need per-tenant or per-user audit dashboards, build them from the formal audit endpoints or from your own host audit store, not from metrics.

> [!WARNING]
> Never trust a `tenantId` sent by the client in a query parameter. The tenant must come from the authenticated host context via the `TenantProvider` SPI.

## 5. Retention and archiving

FileWeft stores audit records in the same persistence schema as document metadata. Retention policy is a host concern:

1. Decide how long to keep audit rows based on your compliance regime.
2. Export older rows to a cold store before deleting them.
3. Do not delete rows while the referenced document is still active unless your policy explicitly allows it.

## FAQ

**Q: Can I write custom events into FileWeft's audit log?**
No. FileWeft logs its own events. Host business events should go to your own audit pipeline.

**Q: Are failed login attempts logged here?**
No. Authentication is a host responsibility; log authentication events in your identity provider or gateway.

**Q: Does the audit log include raw file content?**
No. It contains metadata, actions, and outcomes, never file bytes.
