## 什么属于安全问题

任何可能削弱租户隔离、绕过授权、泄漏对象存储凭据、伪造工作流决策或意外暴露用户数据的问题都应报告。例如：

- 在没有正确 `AuthorizationProvider` 决策的情况下读写其他租户文档的方法。
- 直接信任请求参数中的 `tenantId`，而不是从 `TenantProvider` 获取的代码路径。
- 在日志或错误响应中泄漏签名对象存储 URL、数据库凭据或私钥。
- 可复现的崩溃或资源耗尽，影响共享 Worker 或数据库。
- 绕过生命周期转换，例如未经必要审批即发布文档。

## 如何私密报告

不要在公开 Issue、讨论、PR 评论或社交媒体上披露疑似漏洞。

1. 准备一份最小、可复现的报告。
2. 发送邮件至 **support@icen.ai**，主题前缀使用 `[SECURITY] FileWeft`。
3. 提供维护者可安全联系你的方式，用于协调后续跟进。
4. 在公开披露前，预留合理时间进行调查和修复。

> [!WARNING] 公开披露会让所有人处于风险中
> 在修复可用之前公开分享漏洞，可能导致生产宿主在升级前就被利用。

## 报告中应包含什么

一份好的报告能节省数天往返沟通。尽量包含以下内容：

- 受影响的 FileWeft 版本和模块（如 `fileweft-core`、`fileweft-web-api`、某个 Starter）。
- 部署前提：Spring Boot 代际、数据库、存储后端、认证方式。
- 审阅者可在本地运行的最小复现步骤或概念验证。
- 已观察影响和已知缓解方式。
- 你偏好的安全联系方式，用于协调披露。

```text
主题：[SECURITY] FileWeft - workflow decisions 中疑似租户隔离绕过

受影响版本：ai.icen:fileweft-web-api:0.0.3
模块：fileweft-web-runtime
部署环境：Spring Boot 3.2、PostgreSQL 16、MinIO

摘要：
调用 POST /fileweft/v1/documents/{id}/submit 并传入特定 header 时，
在条件 X 下似乎跳过了 AuthorizationProvider 检查。

复现步骤：
1. 以默认内存授权 Provider 启动样例宿主。
2. 以租户 A 创建文档 D。
3. 发送 submit 请求，header X 设置为 ...
4. 观察到工作流在没有任何授权决策的情况下转换。

影响：
租户 A 的已认证用户可能在未做权限评估的情况下触发生命周期变更。

报告者已知的缓解方式：
无。

联系方式：reporter@example.com（附 PGP 公钥）
```

## 保护敏感证据

移除生产凭据、租户数据、用户标识、对象键、内部端点和私有堆栈跟踪。尽量使用合成夹具。

绝不要在 Issue、日志片段、浏览器录屏或聊天消息中发送以下内容：

- 有效访问令牌或会话 Cookie。
- 数据库连接串或密码。
- 对象存储 access-key/secret-key。
- 真实租户 ID、文档 ID 或用户 ID。
- 私钥或证书材料。

## 披露时间线

FileWeft 维护者遵循协调披露流程：

1. **确认收到** —— 五个工作日内回复，确认已收到报告。
2. **调查** —— 维护者复现问题并评估影响。
3. **修复与验证** —— 准备补丁，在受支持矩阵上测试，并私下分享给你验证。
4. **发布** —— 发布修复版本并记录。
5. **公开披露** —— 宿主有合理升级时间后，发布安全公告并致谢报告者。

> [!NOTE] 一般使用问题也可联系 support@icen.ai
> 使用问题可以使用同一邮箱，但疑似漏洞请明确标记为私密、安全敏感。

## 常见问题

**我会获得 CVE 吗？**
如果问题符合 CVE 条件，维护者会申请 CVE 并把你署名为报告者。

**90 天后我可以自行公开吗？**
项目倾向于协调披露。如果进展停滞，请联系维护者商定合理时间线。

**FileWeft 有漏洞赏金计划吗？**
目前暂无正式赏金计划，但仍感谢并认可负责任的披露。
